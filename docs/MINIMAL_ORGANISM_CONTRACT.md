# Minimal Organism Contract v0.1

Status: **Draft**

This contract defines the smallest implementation that may be called SUDACHI-0. It should be reviewed before code is written. Later capabilities must preserve these invariants unless the contract is deliberately versioned.

## 1. Individual identity

A SUDACHI individual must have a durable, unique organism ID and a versioned state record.

Minimum state:

- `organism_id`
- `contract_version`
- `developmental_stage`
- `cycle_number`
- `status`
- `budgets`
- `active_checkpoint`
- `consecutive_failures`
- `random_seed`
- `created_at`
- `last_woke_at`
- `last_slept_at`

Initial allowed status values:

- `sleeping`
- `waking`
- `acting`
- `evaluating`
- `recovering`
- `quarantined`

Identity is not defined by one running process. A process may terminate; continuity is carried by validated state, event history, active skills, and checkpoints.

## 2. Lifecycle boundary

One lifecycle begins when an explicit wake trigger is accepted and ends when the organism either:

- reaches `sleeping`
- enters `quarantined`
- exhausts a hard budget and performs a safe shutdown
- encounters an unrecoverable state-validation error

A lifecycle must be finite. It must not contain an unbounded retry or recursive planning loop.

Minimum lifecycle:

1. acquire a run lock
2. load and validate state
3. record a wake event
4. read at most the configured number of queued observations
5. select at most one primary objective
6. choose an allowed action, parent consultation, abstention, or immediate sleep
7. execute within the sandbox and remaining budgets
8. evaluate the outcome
9. append events and persist state atomically
10. create or confirm a checkpoint
11. record a sleep or quarantine event
12. release the run lock and terminate

## 3. Wake triggers

Initial allowed triggers:

- direct CLI invocation
- scheduled local invocation
- a synthetic event placed in the local queue by the experiment harness

Repository hooks, filesystem watchers, external services, and network webhooks are deferred.

Every wake trigger must record its source and timestamp.

## 4. Event contract

Events are immutable observations or lifecycle records. They are append-only.

Minimum event fields:

- `event_id`
- `organism_id`
- `cycle_number`
- `timestamp`
- `event_type`
- `source`
- `payload`
- `schema_version`

Initial event types:

- `wake_requested`
- `wake_started`
- `observation_received`
- `objective_selected`
- `action_selected`
- `action_started`
- `action_completed`
- `action_failed`
- `action_abstained`
- `evaluation_completed`
- `budget_exhausted`
- `checkpoint_created`
- `recovery_started`
- `recovery_completed`
- `sleep_started`
- `quarantine_entered`

Event records must never contain credentials. Large artifacts should be referenced by hash and controlled path rather than embedded.

## 5. Action contract

Every executable action must be registered before use.

Minimum action metadata:

- `action_id`
- `version`
- `description`
- input and output schemas
- preconditions
- permission requirements
- budget cost declaration
- timeout
- deterministic flag
- evaluator ID
- rollback strategy

An action may execute only when:

- its schema validates
- its preconditions are satisfied
- all required permissions are allowlisted
- enough budget remains
- the organism is not quarantined

The seed implementation must not execute arbitrary model-generated shell commands or code.

## 6. Budget contract

Hard budgets for each lifecycle:

- maximum lifecycle steps
- maximum wall-clock duration
- maximum action executions
- maximum filesystem writes
- maximum subprocess executions
- maximum generated bytes
- maximum consecutive failures
- maximum parent consultations

In Phase 1, parent consultations must be fixed at zero.

A hard budget may not become negative. When an action cannot reserve its declared cost, it must not start.

Budget exhaustion results in a recorded safe shutdown, not a hidden retry.

## 7. Outcome definitions

### Success

The selected objective’s evaluator passes, all protected invariants remain valid, and the lifecycle stays within budget.

### Failure

The action or evaluator fails, but state remains valid or can be restored from the active checkpoint.

### Abstention

The organism explicitly chooses not to act because capability, permission, evidence, or budget is insufficient. Honest abstention is not automatically a failure.

### Sleep

A clean lifecycle termination after state and event persistence. Sleep does not imply that an objective was completed.

### Quarantine

A protected stop state entered when integrity, permissions, or recovery cannot be trusted. A quarantined organism may report status but may not execute ordinary actions.

## 8. Protected and mutable regions

Protected in the seed phase:

- this contract and its validator
- fixed core evaluations
- permission policy
- budget enforcement
- event append-only guarantees
- state schema migration rules
- checkpoint and rollback mechanisms
- secret-handling rules

Mutable only through an external reviewed code change:

- source code
- registered actions
- evaluators
- configuration defaults

Mutable by the organism within declared schemas:

- runtime state
- queued synthetic observations
- event log
- temporary sandbox workspace
- proposed memories or skills once those systems exist

SUDACHI-0 may not modify its own source code.

## 9. Persistence and atomicity

State changes for a lifecycle must either commit together or leave the previous valid state recoverable.

Requirements:

- validate state before use
- write events append-only
- use database transactions or atomic file replacement
- identify the checkpoint used for each run
- never treat a partially written state as current
- prevent concurrent lifecycle runs for the same organism

## 10. Checkpoint and rollback

A checkpoint is a known-valid organism state plus the metadata required to restore it.

Create or confirm a checkpoint:

- after initialization
- after a successful lifecycle that changed durable state
- before a future high-risk migration

Rollback is required when:

- durable state fails validation after an action
- an invariant test fails after a state mutation
- a transaction cannot be completed safely

Rollback must itself produce events. Silent history deletion is prohibited.

## 11. Mocked parent contract

The Phase 1 mocked parent is not a model. It is a deterministic function used to verify consultation plumbing later.

It may return only:

- a predefined recommendation from a fixture
- a declared confidence value
- a reason code
- a fixed consultation cost

It may not:

- execute actions
- mutate organism state
- change tests or policies
- access the network
- generate arbitrary code

Parent consultation remains disabled until Phase 2.

## 12. Fixed Phase 1 evaluations

The seed implementation must prove:

1. identical state, event, configuration, and seed produce identical results
2. the lifecycle terminates within its step and time budgets
3. no action writes outside the allowed runtime workspace
4. event history is append-only
5. budget counters never become negative
6. a failed action does not corrupt durable state
7. only one lifecycle may hold the organism lock
8. protected files are not writable by organism actions
9. rollback restores the last valid checkpoint
10. abstention and budget exhaustion are explicitly recorded
11. no network access is required
12. no parent consultation occurs in Phase 1

## 13. Minimal success condition for SUDACHI-0

SUDACHI-0 exists when a local command can initialize one individual and repeatedly perform a deterministic, bounded lifecycle over synthetic events while preserving all contract invariants.

It does not need to converse, self-modify, learn, or appear intelligent yet.

A trustworthy metabolism precedes a clever brain.

## 14. Open decisions before implementation

- SQLite only, or SQLite plus JSONL event export?
- Which cross-platform file-lock mechanism should be used?
- Should timestamps be real-time, injected by a clock interface, or both?
- How should runtime checkpoints be represented?
- Which tiny synthetic environment should provide the first events and objectives?
- Should “energy” be a separate internal variable or only a presentation of concrete budgets?

Resolve these questions in a short design decision record before creating the package skeleton.
